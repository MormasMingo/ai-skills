# 文档管理规范

## 概述

本文档管理Skill用于规范项目中的文档创建、更新和维护流程，确保项目文档的一致性、完整性和可维护性，支撑系统的长期演进和团队协作。

## 触发条件

当以下场景出现时，应调用此Skill：
- 用户需要创建或初始化 `.trae/rules/project_rules.md` 文件
- 用户需要创建或更新项目设计文档 (design.md)
- 用户需要初始化 `documents/` 目录及文档管理规范体系
- 用户需要创建或管理 `documents/checkpoints/` 检查点文档
- 新建项目时需要初始化完整文档体系
- 项目迭代中需要自动更新文档内容
- 对话过长或上下文丢失时需要复读检查点

### 3.3 任务跟进机制

#### 3.3.1 文档管理工具使用
- **每次对话开始前**必须查阅 `documents/checkpoints/` 目录下的最新检查点文档
- **任务状态变更时**立即更新检查点文档中的待办任务状态
- **新需求产生时**立即创建或更新检查点文档
- **项目迭代时**同步更新 design.md 和检查点文档

#### 3.3.2 检查点机制执行
- **禁止行为**必须严格遵守，违反可能导致严重后果
- **建议规范**建议遵守，有助于提高质量和效率
- **待办任务**必须可执行、可验证，避免模糊描述
- **后续备忘**记录暂不处理但重要的事项，定期回顾

#### 3.3.3 进度跟踪要求
- 使用 `[x]` 标记已完成任务，`[ ]` 标记待完成任务
- 任务完成后**立即**更新状态，不得延迟
- 复杂任务分解为多个子任务，分别跟踪
- 定期检查并归档已完成的检查点文档

#### 3.3.4 需求管理规范
- 新需求必须记录到检查点文档的"待办任务"中
- 需求变更必须更新 design.md 的"迭代记录"
- 需求优先级必须在检查点文档中明确标注
- 需求完成情况必须在检查点文档中验证确认

#### 3.3.5 方案生成文档化规范
- **凡是需要生成方案的时候**，都必须自动生成文档进行保存
- 方案文档必须包含：问题描述、解决方案、实施步骤、预期结果
- 方案文档必须遵循文档编号体系和元信息规范
- 复杂方案必须分解为多个子文档，分别编号管理
- 方案实施过程中产生的变更必须同步更新到原方案文档
- 方案完成后必须更新检查点文档的状态和结果

#### 3.3.6 文档生成自动化要求
- 生成方案时自动创建对应的文档目录结构
- 自动填充文档元信息（doc-id、version、status、created、updated、author）
- 自动建立文档间的关联关系（引用、依赖、父子文档）
- 自动更新相关文档的引用和索引
- 自动归档旧版本文档，保持文档历史可追溯

### 3.4 方案生成文档化规范

## 二、Documents 文档管理规范

### 2.1 目录结构

```
documents/
├── README.md                              # 文档总览和导航
├── 00-specification/                      # 【通用规范】
│   ├── 00-document-management-specification.md
│   └── 01-checkpoint-management-specification.md
│
├── 01-projects/                           # 【项目文档】
│   └── [project-name]/
│       ├── README.md
│       ├── 00-system/                    # 系统级文档
│       ├── 01-standards/                 # 项目特定规范
│       ├── 02-architecture/              # 架构设计
│       └── 03-modules/                   # 模块设计
│
└── checkpoints/                           # 【检查点文档】
    ├── README.md
    └── YYYY-MM-DD-[序号]-[描述].md
```

### 2.2 文档编号体系

#### 格式
```
[层级]-[模块]-[分类]-[序号]
```

#### 层级定义
| 层级代码 | 层级名称 | 说明 |
|----------|----------|------|
| 00 | spec | 通用规范 |
| 01 | [project] | 项目级文档 |
| 02 | [module] | 模块级文档 |

#### 分类定义
| 分类代码 | 分类名称 | 说明 |
|----------|----------|------|
| 00 | index | 索引类 |
| 01 | requirements | 需求类 |
| 02 | design | 设计类 |
| 03 | implementation | 实现类 |
| 04 | migrations | 迁移类 |
| 05 | changelog | 变更类 |

### 2.3 文档元信息规范

每个规范文档必须以YAML Front Matter开头：

```yaml
---
doc-id: 00-spec-00-001          # 文档唯一编号（必填）
version: 1.0.0                  # 版本号（必填）
status: draft|review|published  # 状态（必填）
created: YYYY-MM-DD             # 创建日期（必填）
updated: YYYY-MM-DD             # 更新日期（必填）
author: [作者]                   # 作者（必填）
reviewer: [审核人]               # 审核人（可选）
tags: [tag1, tag2]              # 标签（可选）
---
```

### 2.4 初始化文档管理规范流程

1. **创建目录结构**
   ```bash
   mkdir -p documents/00-specification
   mkdir -p documents/01-projects
   mkdir -p documents/checkpoints
   ```

2. **创建 00-document-management-specification.md**
   - 定义文档目录结构
   - 定义编号体系
   - 定义元信息规范

3. **创建 01-checkpoint-management-specification.md**
   - 定义检查点文档结构
   - 定义文件命名规范
   - 定义四个标准部分

4. **创建 README.md**
   - 文档体系说明
   - 导航索引

### 2.5 自动更新文档规则

当项目迭代时，自动更新以下文档：

1. **design.md 更新**
   - 新需求追加到"功能需求"章节
   - 架构变更追加到"技术架构"章节
   - 每次更新追加到"迭代记录"

2. **检查点文档更新**
   - 任务完成后立即更新待办状态
   - 新发现的问题追加到"后续备忘"
   - 已完成的任务迁移到"已完成"

3. **规范文档更新**
   - 新规范追加到对应章节
   - 版本号递增
   - 更新日期同步修改

## 三、Checkpoints 检查点管理

### 3.1 检查点目录位置

**固定路径**: `documents/checkpoints/`

### 3.2 检查点文档结构

每个检查点文档必须包含四个部分：

```markdown
# 检查点：[标题]

---
checkpoint-id: YYYY-MM-DD-[序号]
date: YYYY-MM-DD
status: active|completed|archived
---

## 第一部分：禁止行为（必须严格遵守）

- [ ] **禁止** [具体禁止行为]
- [ ] **禁止** [具体禁止行为]

## 第二部分：建议规范（建议遵守）

### 类别1
- [ ] [建议内容]
- [ ] [建议内容]

## 第三部分：待办任务（本次需求）

### 已完成 ✅
- [x] [已完成任务]

### 进行中 🔄
- [ ] [进行中任务]

### 待开始 ⏳
- [ ] [待开始任务]

## 第四部分：后续备忘（暂不处理）

- [后续需要处理的事项]
```

### 3.3 文件命名规范

```
YYYY-MM-DD-[序号]-[简要描述].md
```

示例：
- `2026-02-27-001-初始化项目文档.md`
- `2026-02-27-002-用户模块开发.md`

### 3.4 检查点复读机制

**复读时机**：
- 每次对话开始前
- 对话过长或被压缩后
- 上下文丢失时

**复读流程**：
1. 读取 `documents/checkpoints/` 目录下最新的检查点文档
2. 提取四个部分的核心内容
3. 向用户确认当前任务状态

### 3.5 检查点更新机制

**更新时机**：
- 任务状态变更时（完成/开始）
- 发现新的禁止行为或建议规范
- 产生新的后续备忘事项

**更新流程**：
1. 读取当前检查点文档
2. 更新相应部分的状态
3. 追加新的内容
4. 保存文档

### 3.6 创建新检查点流程

1. **确定文件名**
   - 格式: `YYYY-MM-DD-[序号]-[描述].md`
   - 序号根据当天已有检查点递增

2. **填充四个部分**
   - 禁止行为：基于 project_rules 和当前任务识别
   - 建议规范：基于最佳实践
   - 待办任务：分解本次需求
   - 后续备忘：记录暂不处理的事项

3. **设置初始状态**
   - status: active
   - 所有待办任务标记为 `[ ]`
