## 🎯 规范目标

建立对话检查点文档管理机制，防止因对话过长、压缩或上下文丢失导致的需求遗漏，确保AI助手能够：
1. 随时查阅当前任务的完整需求
2. 了解必须遵守的禁止行为
3. 跟踪任务完成进度
4. 记录后续需要处理的事项

---

## 📝 文件命名规范

### 格式

```
YYYY-MM-DD-[序号]-[简要描述].md
```

### 组成部分

| 部分 | 说明 | 示例 |
|------|------|------|
| YYYY-MM-DD | 日期 | 2026-02-24 |
| 序号 | 当天第几个检查点，3位数字 | 001, 002 |
| 简要描述 | 检查点主题，简短明了 | 文档规范重构 |

### 示例

```
2026-02-24-001-文档规范重构.md
2026-02-24-002-新增检查点机制.md
2026-02-25-001-用户模块开发.md
```

## 📋 文档结构规范

每个检查点文档必须包含以下四个部分：

### 第一部分：禁止行为（必须严格遵守）

**作用**：列出绝对不允许做的事情，违反可能导致严重后果

**内容要求**：
- 使用 `- [ ]` 复选框格式
- 明确、具体、可执行
- 包含代码相关和流程相关的禁止项

**示例**：
```markdown
## 🚫 第一部分：禁止行为（必须严格遵守）

- [ ] **禁止**使用任何形式的模拟数据、虚拟数据、假数据
- [ ] **禁止**提交未经用户确认的代码到生产分支
- [ ] **禁止**删除或修改 `.trae/rules/project_rules.md` 文件
```

### 第二部分：建议规范（建议遵守）

**作用**：推荐的最佳实践，不强制但建议遵循

**内容要求**：
- 使用 `- [ ]` 复选框格式
- 涵盖沟通规范、文档规范、代码规范等
- 有助于提高质量和效率

**示例**：
```markdown
## 💡 第二部分：建议规范（建议遵守）

### 沟通规范
- [ ] 每次回复前检查是否有待办任务需要更新
- [ ] 使用检查点文档验证当前任务状态

### 代码规范
- [ ] 遵循项目现有的代码风格
- [ ] 修改前阅读相关代码了解上下文
```

### 第三部分：待办任务（本次需求）

**作用**：记录本次对话需要完成的任务清单

**内容要求**：
- 使用 `- [ ]` 未完成 / `- [x]` 已完成 格式
- 按状态分组：已完成、进行中、待开始
- 任务描述清晰、可执行
- 包含子任务时使用缩进列表

**示例**：
```markdown
## 📋 第三部分：待办任务（本次需求）

### 已完成 ✅
- [x] 设计检查点文档格式
- [x] 创建检查点目录

### 进行中 🔄
- [ ] 更新项目规则文件
  - [ ] 补充检查点查阅规则
  - [ ] 说明查阅时机

### 待开始 ⏳
- [ ] 创建检查点管理规范
```

### 第四部分：后续备忘（暂不处理）

**作用**：记录当前不处理但需备查的内容

**内容要求**：
- 使用 `- [ ]` 复选框格式
- 分为"待后续考虑"和"备查信息"
- 包含相关路径、编号等关键信息

**示例**：
```markdown
## 📦 第四部分：后续备忘（暂不处理）

### 待后续考虑
- [ ] 考虑开发检查点文档自动生成脚本
- [ ] 考虑增加检查点文档的关联关系

### 备查信息
- **检查点目录**: `documents/05-checkpoints/`
- **命名规范**: `YYYY-MM-DD-[序号]-[描述].md`
```

## 🔄 使用流程

### 1. 创建检查点

**时机**：对话开始时，或需求变更时

**步骤**：
1. 根据命名规范创建新文件
2. 填写文档头部元信息
3. 完整填写四个部分的内容
4. 更新 `README.md` 中的当前检查点列表

### 2. 更新检查点

**时机**：任务状态变更时

**步骤**：
1. 找到当前检查点文档
2. 更新待办任务的完成状态
3. 如有新增禁止项或建议，补充到相应部分
4. 更新变更记录

### 3. 查阅检查点

**时机**：
- 每次对话开始前
- 对话过长或被压缩后
- 不确定当前任务状态时

**步骤**：
1. 打开 `documents/05-checkpoints/` 目录
2. 找到最新的检查点文档（按日期和序号）
3. 完整阅读四个部分的内容
4. 确认当前任务状态和约束条件

### 4. 归档检查点

**时机**：对话结束，所有任务完成

**步骤**：
1. 将检查点文档状态改为 `completed`
2. 更新 `README.md` 中的状态
3. （可选）移动到归档目录

## 📊 质量检查清单

### 创建时检查

- [ ] 文件命名符合规范
- [ ] 包含完整的四个部分
- [ ] 禁止行为明确具体
- [ ] 待办任务可执行
- [ ] 已更新 README 索引

### 更新时检查

- [ ] 任务状态及时更新
- [ ] 变更记录完整
- [ ] 无遗漏的待办项

### 查阅时检查

- [ ] 确认是最新检查点
- [ ] 理解所有禁止行为
- [ ] 明确当前任务状态

---

## 🛠️ 工具支持

### 推荐实践

1. **模板使用**：创建新检查点时复制模板
2. **快速查阅**：使用编辑器书签功能标记当前检查点
3. **状态同步**：任务完成立即更新，避免遗漏

### 自动化建议

- 考虑开发脚本自动生成检查点文档框架
- 考虑在对话开始时自动提示查阅检查点
