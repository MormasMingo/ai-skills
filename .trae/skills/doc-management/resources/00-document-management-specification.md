## 🎯 规范目标

建立标准化的文档管理体系，确保系统架构、模块设计、开发规范等文档的完整性、一致性和可维护性，支撑系统的长期演进和团队协作。

**核心目标**：
1. **可追溯** - 完整记录需求演进、设计决策、实现过程
2. **可理解** - 分层分模块描述，从架构到细节逐层深入
3. **可复现** - 文档足以支撑系统级别的重建和复现
4. **可维护** - 统一规范、统一结构、统一命名

> **说明**：本文档为**通用基础规范**，不涉及具体业务系统。各业务系统应基于此规范建立补充说明。

## 📝 文档元信息规范

### 文档头部模板

每个文档必须以YAML Front Matter开头：

```yaml
---
doc-id: 02-user-02-001          # 文档唯一编号（必填）
title: 文档标题                  # 文档标题（必填）
description: 文档简要描述       # 文档描述（必填）
version: 1.0.0                  # 文档版本（必填）
status: published               # 状态：draft/review/published/archived（必填）
created: 2026-02-04             # 创建日期 YYYY-MM-DD（必填）
updated: 2026-02-09             # 更新日期 YYYY-MM-DD（必填）
author: 架构团队                 # 作者（必填）
reviewer: 技术负责人             # 审核人（必填）

# 关联关系（必填）
relations:
  parent: 02-user-00-001        # 父文档编号
  children:                     # 子文档编号列表
    - 02-user-02-002
    - 02-user-02-003
  related:                      # 相关文档编号列表
    - 02-user-03-001
    - 01-贵州河湖-02-001
  
# 代码关联（可选）
code:
  version: 1.2.0                # 对应代码版本
  branch: feature/user-v2       # 对应分支
  commit: abc123                # 对应提交哈希
  path: /backend/module-user    # 代码路径（相对项目根目录）

# 需求关联（可选）
requirements:
  - REQ-USER-001
  - REQ-USER-002

# 待解决问题（可选）
pending-issues:
  - issue: ISSUE-USER-001
    description: 性能优化
    priority: high|medium|low
    planned-date: 2026-03-01

# 标签（可选）
tags:
  - 用户管理
  - 架构设计
  - API设计
---
```

### 文档状态定义

| 状态 | 说明 | 转换规则 |
|------|------|----------|
| draft | 草稿，内容不完整 | → review |
| review | 审核中，等待审批 | → published / draft |
| published | 已发布，正式使用 | → archived |
| archived | 已归档，不再维护 | - |

## 📂 文件命名规范

### 命名原则

1. **全小写**：所有文件名使用小写字母
2. **连字符分隔**：单词间使用连字符 `-` 分隔
3. **语义化**：名称准确反映文档内容
4. **版本化**：重要文档需要版本标识

### 命名格式

```
# 正式文档
[序号]-[描述].md

# 示例
00-overview.md
01-requirements.md
database-design.md
api-design-v2.md

# SQL脚本
[序号]-[描述]-[日期].sql

# 示例
01-init-database.sql
02-migration-20260206.sql
```

### 禁止事项

- ❌ 使用大写字母：`APIDesign.md`
- ❌ 使用下划线：`api_design.md`
- ❌ 使用中文：`接口设计.md`
- ❌ 使用空格：`api design.md`
- ❌ 重复命名：`api-design.md` 和 `API-DESIGN.md`

## 🚨 TODO 管理规范（P0级别）

### 绝对禁止
- **严禁**在关键业务流程中遗留 TODO/FIXME/XXX 标记
- **严禁**返回空数组/空对象/占位符而不实现实际逻辑
- **严禁**用 TODO 替代实际业务校验逻辑

### 违规后果
- 导致业务流程中断或数据错误
- 造成严重的数据冗余或数据丢失
- 系统看似正常运行，实际关键环节缺失

### 代码提交前强制检查清单
```markdown
## 代码审查清单（提交前必须完成）

### 逻辑完整性
- [ ] 所有参数是否正确解构和传递
- [ ] 调用方和实现方的参数是否一致
- [ ] 多选/单选等边界情况是否处理
- [ ] 异常情况和错误处理是否完善

### TODO 检查
- [ ] 搜索全代码，确认无 TODO/FIXME/XXX 遗留
- [ ] 关键流程无空实现或占位符
- [ ] 所有业务逻辑已完整实现

### 数据流检查
- [ ] 输入数据校验是否完整
- [ ] 输出数据格式是否正确
- [ ] 数据转换逻辑是否正确
```

### 合规做法
1. **必须实现** - 关键流程代码必须完整实现，不能留空
2. **临时标记** - 如必须写 TODO，必须同时：
   - 创建跟踪任务到检查点文档
   - 明确标注影响范围和风险等级
   - 设定解决期限
3. **提交前检查** - 每次提交前必须搜索并清理所有 TODO

## 📋 附录

### 附录A：Git 提交规范

#### 提交信息格式
```
<type>: <subject>

<body>
```

#### Type 类型
| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: add skill scanner module` |
| `fix` | 修复bug | `fix: resolve keyboard input issue` |
| `refactor` | 重构代码 | `refactor: optimize UI rendering logic` |
| `docs` | 文档更新 | `docs: update project rules` |
| `style` | 代码格式 | `style: format ui.py` |
| `test` | 测试相关 | `test: add scanner test cases` |
| `chore` | 构建/工具 | `chore: update build scripts` |

#### Subject 规范
- 使用**英文**，首字母小写
- 不超过 50 个字符
- 使用**动词开头**，描述做了什么
- 结尾**不加句号**

**正确示例**：
- `feat: add interactive skill selection UI`
- `fix: resolve arrow key handling on Windows`
- `refactor: optimize terminal width calculation`

**错误示例**：
- ❌ `feat: added new feature` (过去时)
- ❌ `fix: bug fixed.` (有句号)
- ❌ `update` (无类型，无描述)
- ❌ `refactor` (无具体描述)

#### Body 规范（可选）
- 当修改较复杂时，添加 body 说明
- 描述**为什么修改**和**修改了什么**
- 每行不超过 72 个字符

#### 提交时机
- **每个独立功能/修复**单独提交
- **提交前**检查修改内容
- **提交信息**必须准确描述修改内容

#### 命令行提交方式
```bash
# 正确方式 - 使用双引号包裹
git commit -m "feat: add skill scanner module"
git commit -m "fix: resolve keyboard input issue"

# 错误方式 - 单引号在某些环境下可能解析失败
git commit -m 'feat: add skill scanner'  # 可能出错

# 错误方式 - 不带引号，空格会导致解析错误
git commit -m feat: add skill scanner    # 错误！
```
